### 设计讨论

- **Reward**：确定采用 outcome-based，不做 verifier-based。理由：verifier 是额外成本，偏僻场景下 verifier 可能不存在
- **核心机制**：每次 rollout 记录 notes，notes 就是 experience buffer。后续 agent 读取 notes 在 context 中调整行为——权重没变，策略变了
- **RL 粒度**：Task 是 rollout 单位，Expert 是多个 task context 并集的涌现结果
- **信息沉淀路径**：notes（task 细节）→ doc（跨 task 知识并集）→ SOP（结晶策略），且压缩过程本身也是 task，用同一套机制学习。系统只有一种原语：task rollout + notes 记录
- **两种模式**：
  - Human-driven：人在 loop 里做 verifier，meta-agent 已经实现得比较好
  - Autonomous：agent 自己跑 rollout，自己拿 outcome signal。这是 bfrl 要重点解决的
- **下一步待定义**（autonomous 模式）：

  2. Outcome signal 从哪来（没有人的时候）
  3. Rollout loop 怎么跑
