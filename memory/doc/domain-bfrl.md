# 特定领域 BFRL 方法

## 核心判断

要把 BFRL 做到“特定领域好用”，关键不是往 prompt 里塞更多领域知识，也不是先造一个通用 agent 框架，而是把这个领域改造成一个**可反复 rollout、可观测 outcome、可沉淀 context** 的闭环。

换句话说，特定领域 BFRL 的对象不是“模型”，而是：

- 任务分布
- outcome 信号
- context 组成方式
- 经验沉淀方式
- 后续 rollout 对历史经验的利用方式

传统 RL 在权重空间做策略优化；特定领域 BFRL 则是在上下文空间做策略优化。

## 先决条件：什么样的“领域”适合做 BFRL

不是所有领域都适合同一种 BFRL。最适合先做的，是满足下面四个条件的领域：

1. **任务有重复性**
   - 同一类任务会反复出现，task 之间有足够高的重叠
   - 否则 notes 无法沉淀成 expert context

2. **环境可操作**
   - Agent 能通过工具真实接触这个领域，而不是只能空想
   - 例如读文档、查数据库、跑脚本、调 API、调用测试器、访问监控系统

3. **结果可观测**
   - 能拿到相对明确的 outcome signal
   - 最好来自环境本身，而不是额外训练一个 verifier

4. **反馈周期够短**
   - 最好在一次 rollout 或少数几次 rollout 内看到结果
   - 如果 reward 要几个月后才知道，autonomous BFRL 很难跑起来

因此，领域切分必须足够窄。不要上来就做“法律”“医疗”“运维”“研发”这种大领域，而要切成：

- 某类合同审阅
- 某类工单分诊
- 某类故障排查
- 某类代码修复
- 某类报告生成与校验

领域越窄，task 重叠越高，expert context 越容易涌现。

## 特定领域 BFRL 的正确目标

对于特定领域，不应该把目标写成“让 agent 学会这个领域”，而应该写成：

> 让 agent 在这个领域中，通过反复 rollout 和 outcome feedback，逐步学会如何构造更有效的 context。

这里被优化的不是权重，而是五件事：

1. 当前 task 应该读哪些历史经验
2. 当前 task 应该调用哪些工具
3. 当前 task 应该遵循哪个 SOP 或计划模板
4. 当前 task 结束后，经验应该沉淀到哪一层
5. 哪些情况需要升级给人，而不是继续自动 rollout

## 特定领域 BFRL 的最小闭环

要做成 autonomous BFRL，至少要把仓库里尚未定义清楚的三个问题补上：

1. Task 从哪来
2. Outcome signal 从哪来
3. Rollout loop 怎么跑

### 1. Task 来源：不要只靠真实任务

特定领域 BFRL 的 task 来源至少要有三类，而且要混合：

#### A. 真实任务

来自生产环境或真实工作流：

- 用户工单
- backlog 事项
- 实际请求
- 线上故障
- 待处理文档/代码/报告

这是最重要的主分布，保证训练方向和真实价值一致。

#### B. 失败重放任务

来自历史 notes 里失败或部分失败的 case：

- 曾经出错的任务
- 曾经升级给人的难例
- 反复出现的边界 case
- 高成本失败模式

这相当于 prioritized replay。对特定领域来说，它比均匀重放更重要，因为领域内真正拉开水平差距的，往往就是少数高频难点。

#### C. 领域合成任务

不是凭空造题，而是围绕真实任务做局部扰动：

- 改参数
- 改约束
- 改输入格式
- 改异常条件
- 改边界条件

例如：

- 把“正常合同条款审阅”扰动成“同类条款但多一条例外约束”
- 把“单机部署失败”扰动成“网络波动 + 版本漂移”
- 把“SQL 修复”扰动成“相同 schema、不同脏数据模式”

这类 task 的作用是扩展局部邻域，避免 BFRL 只记住历史样本而不会泛化。

### 2. Outcome signal：必须来自领域环境本身

仓库已经明确要求 outcome-based reward，而不是 verifier-based。对于特定领域，关键不是“有没有 reward”，而是“reward 能否直接从环境里拿到”。

最好的做法是设计一个**领域 outcome harness**。

这里补一个术语说明：`Outcome Harness` 不是我目前看到的外部标准固定术语，它是建立在近来流行起来的 `Harness Engineering` 语境上的一个项目内延伸说法。它专指 harness 中那一部分负责：

- 从领域环境读取任务最终状态
- 把最终状态转换成可用的 outcome signal
- 为后续 rollout / consolidation 提供 success、quality、cost、risk、escalation 等判定

因此，`Outcome Harness` 不是另一个大模型 verifier，也不是整个 agent harness，而是更窄的一层：**面向 outcome 的环境判定与反馈层**。

它不一定输出单一标量，完全可以输出 outcome card：

- `success`: 成功 / 失败 / 部分成功
- `quality`: 质量指标
- `cost`: 时间、token、金钱、资源占用
- `risk`: 是否触发安全/合规边界
- `escalation`: 是否最终需要人介入

但注意：这个 harness 仍然应该是环境规则，不是另一个大模型 verifier。

#### 不同领域的 outcome 来源

1. **可执行任务**
   - 测试是否通过
   - benchmark 是否达标
   - 脚本是否成功
   - SQL 是否返回正确结果
   - 部署是否恢复服务

2. **流程型任务**
   - 工单是否闭环
   - SLA 是否满足
   - 是否命中正确路由
   - 后续是否被人工打回

3. **内容型任务**
   - 是否满足格式/约束检查
   - 是否被下游系统接受
   - 是否通过抽样审计
   - 是否被用户/专家接受

如果一个领域拿不到环境 outcome，只能拿“模型自己觉得答得不错”，那就不是好的 autonomous BFRL 领域。

### 3. Rollout loop：不是一次问答，而是上下文策略搜索

特定领域 BFRL 的 rollout 不应理解成“一次 prompt -> 一次回答”，而应理解成：

1. 选一个 task
2. 组装本次 context
3. 允许 agent 用工具探索
4. 产出结果
5. 用环境 outcome 评分
6. 把经验写回上下文体

其中真正被优化的是“组装本次 context”的方式。

一个合理的 rollout loop 可以写成：

1. 从任务池取出一个 task
2. 检索最相关的 notes、doc、SOP
3. 组装 task context
4. agent 执行任务并调用工具
5. environment/harness 返回 outcome
6. agent 生成本次 rollout note
7. 判断经验应沉淀到 notes、doc、SOP、还是 outcome harness 修订
8. 更新任务池优先级，进入下一次 rollout

## 关键方法：上下文信用分配

这是我认为仓库里还没有显式说出来、但对特定领域 BFRL 最关键的一步。

传统 RL 的难点之一是 credit assignment：奖励信号来了，到底该更新哪些权重。

在 BFRL 里，难点变成：

> 一次成功或失败之后，经验到底应该改写哪一层 context？

对于特定领域，这个问题必须明确，否则 notes 只会越堆越多，而不会真正形成策略改进。

一次 rollout 之后，经验通常有五种去向：

1. **写入 note**
   - 这次经验是局部的、一次性的、还没验证复用性

2. **更新 doc**
   - 这次经验暴露了跨 task 共享的领域知识
   - 例如概念边界、约束解释、因果关系、常见坑

3. **提炼 SOP**
   - 这次经验验证了一个稳定可复用的操作策略
   - 例如固定排查顺序、固定决策流程、固定工具调用序列

4. **修订检索/组装策略**
   - 失败不是因为不会做，而是因为没把正确 context 读进来
   - 例如应该优先读最近失败案例，而不是最长的 doc

5. **修订 outcome harness**
   - 失败其实来自 reward 定义不对
   - 例如系统把“快速完成”当成功，但领域更看重“低风险完成”

这就是 BFRL 的 credit assignment。不是调权重，而是调上下文体的不同层次。

## 领域专用 context stack 应该怎么设计

特定领域不应该只有一个大 prompt，而应该有分层 context：

### 第 0 层：运行时基础约束

- 角色
- 安全边界
- 工具说明
- 何时必须升级给人

### 第 1 层：当前 task context

- 当前输入
- 当前目标
- 本次特有约束
- 终止条件

### 第 2 层：领域 doc

- 核心概念
- 常见约束
- 领域术语
- 因果模型
- 常见失败模式

### 第 3 层：SOP

- 已验证的处理流程
- 决策树
- 排查顺序
- 默认策略

### 第 4 层：检索到的近邻 notes

- 相似任务成功案例
- 相似任务失败案例
- 最近被升级的边界 case

注意：真正优化的是第 3、4 层的选取和组合，而不是把所有历史内容都塞进上下文。

## Notes 不能只是日志，而要能被检索和利用

仓库明确反对把知识强制压成低维 key-value，这个判断是对的。但要让特定领域 BFRL 真正跑起来，notes 至少还需要一层轻量索引。

正确做法不是把 notes 主体结构化，而是：

- notes 正文保持自然语言
- 额外维护少量元信息用于检索

例如：

- task family
- 领域标签
- 涉及工具
- outcome
- 失败模式
- 是否升级给人

这样不会压扁表达空间，但能让后续 rollout 更准确地取到相关经验。

## 特定领域 BFRL 的探索方式

不调权重并不意味着不能探索。探索发生在 context 层。

特定领域里最有效的探索通常有四种：

1. **检索探索**
   - 对同一 task 尝试不同的 notes/SOP 组合

2. **计划探索**
   - 对同一 task 尝试不同的处理顺序或工具调用顺序

3. **提示探索**
   - 对同一 task 尝试不同的任务 framing

4. **任务探索**
   - 对真实任务周围做局部扰动，扩展 task 分布

对高价值任务，可以并行跑多个 rollout 分支，再用环境 outcome 选优。胜出分支写成功经验，落败分支写失败经验。这样就把 test-time search 直接纳入了 BFRL。

## 从 task 到 expert：特定领域最重要的是压缩

如果只堆 notes，不做压缩，系统会很快失效。特定领域 BFRL 的关键不是“记住全部”，而是“不断把 task-level 经验压成 expert-level context”。

正确路径仍然是：

- notes：单次 rollout 的细节
- doc：跨 task 的稳定认知
- SOP：被验证的可执行策略

但对于特定领域，应该增加明确的升级规则：

### 从 note 升级到 doc 的条件

- 同一类结论在多个 task 中重复出现
- 结论不依赖某个具体输入
- 结论能改变后续很多 task 的判断

### 从 doc 升级到 SOP 的条件

- 已经形成稳定步骤
- 在多个 case 中被验证有效
- 起始条件和结束条件足够清楚

### 从 SOP 降级回 doc 或 note 的条件

- 新 case 频繁打破原流程
- 说明这个“策略”其实还没稳定

没有这套升降级机制，特定领域 BFRL 很容易退化成日志堆积。

## 人在特定领域 BFRL 里的作用

autonomous 不等于不要人。人在这个系统里至少做四件事：

1. 选定领域边界
2. 定义或审核 outcome harness
3. 决定何时把经验升格为 SOP
4. 处理价值冲突、优先级冲突和高风险边界

也就是说，人在 BFRL 里主要负责 reward 设计边界和策略制度边界，而不是每次 task 都下场做 verifier。

## 建议的实现顺序

如果现在真的要做“特定领域 BFRL”，我建议按下面顺序，而不是先追求通用框架：

### 阶段 1：选一个 outcome 最清楚的窄领域

优先选择：

- 能调用工具
- 能自动拿 outcome
- 任务会重复
- 失败成本可控

例如代码修复、配置排查、固定格式文档处理、SQL 修复、工单路由。

### 阶段 2：先做 outcome harness

没有稳定 outcome，就没有 RL。先不要急着造 notes 系统，先确认：

- 成功怎么定义
- 失败怎么定义
- 部分成功怎么定义
- 成本和风险怎么算

### 阶段 3：再做 task 池

至少先建三层：

- 真实任务
- 失败重放
- 合成扰动

### 阶段 4：接上最小 rollout loop

实现最小闭环：

- 取 task
- 读 context
- 执行
- 评估 outcome
- 写 note

### 阶段 5：再做压缩和检索优化

当 notes 积累到一定程度后，再加：

- notes 检索
- doc 提炼
- SOP 提炼
- replay 优先级

### 阶段 6：最后才做多模型和基模无关

仓库里已经写得很清楚：先默认单模型运行。等单模型 autonomous BFRL 跑顺后，再考虑：

- 混合生成
- 跨模型读取 notes
- 更中性的 notes 分布

## 一个简单结论

对于特定领域，最可行的方法不是“做一个通用 BFRL 框架，再套到领域上”，而是：

> 先把特定领域变成一个有 task 分布、有环境 outcome、有经验沉淀路径的上下文闭环，再让 BFRL 在这个闭环里发生。

所以真正的方法论可以压成一句话：

> **Domain-specific BFRL = outcome-grounded task loop + notes/doc/SOP compression + context credit assignment**

如果要继续往下做，下一步最值得写的不是更大的理论，而是一个具体领域的 `task schema`、`outcome harness` 和 `rollout loop` 原型。
