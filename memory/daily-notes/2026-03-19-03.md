### `/agents` 正式调度器收口与接口整理

- 本轮首先对齐了一个关键方向：`agents/main.py` 不再作为“手动调试单个 agent 的脚本”，而是作为**正式的总调度入口**存在。
- 当前公开入口已经收敛为：
  - `python agents/main.py`
  - `python agents/main.py --config agents/config.yaml`
- `--help` 里只保留了 `--config`，不再暴露 `run --agent ...`、`watch --agent ...`、`watch-all`、`workspace-root`、`agents-dir`、`max-runs` 之类会把系统重新拉回“半调试、半正式”状态的参数。
- 共享运行参数已经从 Python 常量迁移到 `agents/config.yaml`，当前主要包括：
  - `watch`
  - `sleep_seconds`
  - `memory_root`
  - `default_model`
  - `agent_models`
- `main.py` 现在会输出人类可读的 monitor 行，至少能看见：
  - 载入了哪个 config
  - 当前是不是 watch 模式
  - agent 执行顺序
  - 某个 agent 是被跳过还是被启动
  - 启动时使用的 runner 和 model
  - 完成耗时
  - 本次有哪些 runtime artifact 变化
  - 没有变化时是否正在 sleep

### 本轮完成的核心修正

- 把统一配置的职责从代码常量移到了 `agents/config.yaml`。
- 让 `main.py` 的 CLI 只保留配置文件入口，避免用户还得记住一堆不该手写的参数。
- 保留了正式系统的三段链路顺序：
  1. `note-relation`
  2. `doc-maintenance`
  3. `sop-promotion`
- 保留了“是否需要启动某个 agent”由 watched inputs 指纹决定的机制，而不是每次都强制全跑。
- 为 agent 运行增加了 runtime artifact 校验：
  - 当某个 formal agent 因输入变化而被唤醒时，调度器会检查它是否真的改动了 `runtime/work-orders/*.json` 或 `runtime/state/runtime-state.json`。
  - 这能避免“表面上 session 成功结束，但其实没有落盘任何机器可消费产物”的假成功。
- 修复了一个重要兼容性问题：
  - `memory_root` 现在允许是**仓库外的绝对路径**。
  - 原本 `main.py` 只适合对 workspace 内部相对路径做 glob 指纹计算；现在已经支持对绝对路径 `memory_root` 做 watched-input 扫描和指纹计算。

### 当前正式系统架构

#### 1. 外层：总调度器

当前正式入口是 `agents/main.py`。

它负责：

- 读取 `agents/config.yaml`
- 载入所有 `agents/*/agent.json`
- 按 `order` 排序 agent
- 对每个 agent 的 `wakeup.watch_globs` 计算输入指纹
- 判断该 agent 是 `skip` 还是 `launch`
- 将任务交给指定的 runner
- 校验 formal agent 是否真的写出了 runtime artifact
- 在 watch 模式下继续轮询

它**不负责**：

- 代替 agent 做领域判断
- 硬编码指定“本轮必须处理哪一篇 note”
- 直接做 doc / sop 写作

也就是说，`main.py` 是 orchestration layer，不是 domain reasoning layer。

#### 2. 中层：agent 定义层

每个 agent 位于 `agents/<agent-id>/` 下，当前标准组成是：

- `agent.json`
- `system.md`
- `runtime/`

`agent.json` 负责声明：

- agent 身份和顺序
- task 的目标与输入
- watched inputs
- launcher 类型与其私有配置
- runtime 路径约定

`system.md` 负责声明：

- agent 的角色
- world model
- hard boundaries
- mission
- working procedure
- output contract
- success criteria

#### 3. 下层：runner 层

runner 是启动后端，不是 agent 本体。

当前只有一个已接入的 runner：

- `codex`

其职责是：

- 接收 scheduler 组装好的 generic execution request
- 在自己的私有 workspace 下启动真实 session
- 把执行结果重新包装成统一的 `ExecutionResult`

### 当前 `notes -> doc -> sop` 的正式链路

#### `note-relation`

输入：

- `memory/daily-notes/*.md`

输出：

- `agents/note-relation/runtime/work-orders/*.json`
- `agents/note-relation/runtime/state/runtime-state.json`

职责：

- 判断新 note / 变更 note
- 做 note relation assignment
- 决定 doc routing
- 判断是否存在 SOP candidate signal

#### `doc-maintenance`

输入：

- `agents/note-relation/runtime/work-orders/*.json`

输出：

- `memory/doc/` 中的真实文档改动
- `agents/doc-maintenance/runtime/work-orders/*.json`
- `agents/doc-maintenance/runtime/state/runtime-state.json`

职责：

- 消费上游 note-relation work order
- 做最小 doc patch 或必要的新 doc 创建
- 为下游 SOP 提升写出 machine-readable work order

#### `sop-promotion`

输入：

- `agents/doc-maintenance/runtime/work-orders/*.json`

输出：

- `memory/sop/` 中的真实 SOP 改动
- `agents/sop-promotion/runtime/work-orders/*.json`
- `agents/sop-promotion/runtime/state/runtime-state.json`

职责：

- 消费上游 doc-maintenance work order
- 判断 `promote-new-sop` / `update-existing-sop` / `defer`
- 只把稳定、可执行的知识写成 SOP

### Codex Agent 是如何被启动的

当前正式启动链路如下：

1. 用户运行 `python agents/main.py --config agents/config.yaml`
2. `main.py` 读取 config，并加载所有 `agent.json`
3. `main.py` 根据 `wakeup.watch_globs` 决定某个 agent 是否需要启动
4. 对于需要启动的 agent，`main.py` 组装 `TaskDefinition` 与 `ExecutionRequest`
5. `main.py` 根据 `launcher.kind` 调用 `get_runner(kind)`
6. 当 `kind == "codex"` 时，进入 `agents/runners/codex/runner.py`
7. `CodexRunner` 将 task 和 launcher config 组装成 `TaskSpec + args`
8. `CodexRunner` 调用 `build_adapter(args)`，构造 `CodexCLIAdapter`
9. `CodexCLIAdapter` 内部通过：
   - `codex -m <model> app-server --listen stdio://`
   启动 Codex app-server
10. 随后调用：
   - `thread/start`
   - `turn/start`
   把 prompt 发送给 Codex
11. 运行过程中持续监听：
   - token usage 更新
   - command execution
   - final answer
12. 当 turn 完成后，runner 返回统一的 `ExecutionResult`
13. `main.py` 最后写入 scheduler state，并输出 monitor 行和 JSON result

### 当前给 Codex 的“指令”由哪些层组成

当前并不是只有一个 prompt，而是多层叠加：

#### 1. `system.md`

这是 agent-specific operating contract。

它定义：

- 身份
- 角色边界
- 禁止行为
- 任务步骤
- 输出契约
- 成功标准

这是最接近“你是谁、你该怎么做”的一层。

#### 2. `agent.json` 里的 task 定义

当前 task 层主要包括：

- `task.title`
- `task.goal`
- `task.system_prompt_path`
- `task.memory_paths`
- `task.allow_file_edits`

这里定义的是：

- 本轮任务叫什么
- 目标是什么
- 先读哪些文件
- 是否允许写文件

#### 3. `main.py` 动态补充的运行时上下文

`main.py` 会把这些动态信息附加给 agent：

- `Agent home`
- `Shared memory root`
- `Daily notes dir`
- `Doc dir`
- `SOP dir`
- `Runtime state dir`
- `Runtime work-orders dir`
- `Runtime reports dir`
- `Runner kind`
- `Runner model`
- `Runner workspace dir`
- `Wake reason`
- `Dependencies`
- `Upstream work-order dirs`
- `Wakeup watch globs`

这层让 agent 知道“我这次是为什么被叫醒的、该去哪里找输入和输出目录”。

#### 4. 通用 Codex wrapper prompt

`agents/lib/codex_session_runner.py` 中的 `format_codex_trace_prompt()` 还会统一补充：

- 只做当前任务，不要扩题
- 先读指定文件再行动
- 如果任务要求产物，就要先落盘再汇报
- 这是 UTF-8 中文仓库
- PowerShell 读取 UTF-8 文件时应显式指定编码
- 最终回答必须简短，并说明改了哪些文件或产物

这层是对所有 Codex agent 共用的操作性约束。

### 如果以后要新增一个 runner，如何操作

当前建议步骤：

1. 在 `agents/runners/<new-kind>/runner.py` 新建一个 runner
2. 实现 `RunnerAdapter` 协议，也就是：
   - 提供 `kind`
   - 实现 `run(request: ExecutionRequest) -> ExecutionResult`
3. 在 runner 内部：
   - 使用 `request.task` 读取任务定义
   - 使用 `request.launcher_config` 读取 launcher 私有配置
   - 使用 `request.runner_workspace_dir` 作为 runner 私有工作目录
4. 返回统一的 `ExecutionResult`
5. 在 `agents/runners/registry.py` 里注册：
   - `if normalized == "<new-kind>": return <NewRunner>()`
6. 在目标 agent 的 `agent.json` 中把：
   - `launcher.kind`
   改成新 runner 名称

新增 runner 时应遵守的边界：

- 不要让 runner 直接承担 agent 的领域职责
- 不要把知识写进新的长期 memory 层
- runner 私有文件应只落在 `runtime/launchers/<kind>/`
- runner 只负责“怎么执行”，不负责“该做什么”

### 如果以后要新增一个 agent，如何操作

当前建议步骤：

1. 新建目录：
   - `agents/<new-agent-id>/`
2. 新建：
   - `agent.json`
   - `system.md`
3. 在 `agent.json` 中定义：
   - `id`
   - `name`
   - `order`
   - `task`
   - `dependencies`
   - `wakeup.watch_globs`
   - `launcher.kind`
   - `launcher.config`
4. 让 `task.memory_paths` 指向它真正需要先读的文档
5. 在 `system.md` 中写清楚：
   - 它是谁
   - 它的 hard boundaries
   - 它的 output contract
   - 它成功时必须落盘什么 artifact
6. 选择它在链路中的顺序：
   - 通过 `order`
   - 以及 `dependencies`
7. 如有必要，在 `agents/config.yaml` 的 `agent_models` 中给它单独配置模型

当前加新 agent 时一个**必须记住的实现细节**：

- `main.py` 里现在有一个 `artifact_contract_agents` 常量，当前只列了：
  - `note-relation`
  - `doc-maintenance`
  - `sop-promotion`
- 如果新 agent 也属于 formal chain，并且应该被 scheduler 强制校验 runtime artifact 是否落盘，那么还需要把它加入这份列表。

也就是说：

- 新增 agent 已经可以通过新建 `agents/<id>/agent.json` 被自动发现
- 但如果希望它也享受“formal artifact enforcement”，还需要同步更新 `main.py`

这是当前架构里一个还没有外部化到 config 的点。

### 当前系统的几个重要设计结论

#### 1. 单入口是正确方向

正式系统应该只有一个外部入口：

- `python agents/main.py --config ...`

而不是把用户拉回“手动运行单个 agent”的模式。

#### 2. 指令应该分层，而不是堆到一个大 prompt 里

当前分层是合理的：

- config 层：共享运行设置
- `agent.json` 层：任务和 launcher 声明
- `system.md` 层：角色与契约
- `main.py` 层：动态运行上下文
- runner wrapper 层：统一执行约束

#### 3. runner 必须是后端，不是架构本体

`codex` 只是当前的 launcher adapter，不应该反过来塑造整个系统结构。

#### 4. formal chain 的成功标准必须是“落盘产物”

如果一个 agent session 结束了，但没有写出：

- work order
- runtime state
- 或要求的 doc / sop 改动

那它就不应被视为真正成功。

### 本轮对后续最有价值的沉淀

这次最重要的不是某一段具体代码，而是下面这些约束终于被拉齐了：

- 正式入口只保留总调度
- 配置从代码迁到 config 文件
- monitor 输出可见
- 外部绝对路径 `memory_root` 可支持
- runner / agent / scheduler 的职责边界更清楚了
- “如何改指令、如何加 runner、如何加 agent” 已经可以按固定接口操作，而不必再靠临时口头约定

### 一个直接可用的压缩结论

当前 `/agents` 系统已经可以压缩成一句话：

> `main.py` 作为唯一正式入口，根据 config 和 watched inputs 调度整条 `daily-notes -> doc -> sop` 链；agent 负责领域判断，runner 负责执行后端，所有后续扩展都应围绕 `agent.json`、`system.md`、`ExecutionRequest/ExecutionResult` 和 runner registry 这几层接口进行。

如果后续要把这批内容从 session 经验提升为稳定文档，最值得上升到 `memory/doc/` 的方向包括：

- 正式 `/agents` 运行时架构快照
- runner adapter contract 的稳定版本
- 新 agent / 新 runner 的扩展 SOP
