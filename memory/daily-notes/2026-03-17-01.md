### 设计讨论

- **Reward**：确定采用 outcome-based，不做 verifier-based。理由：verifier 是额外成本，偏僻场景下 verifier 可能不存在
- **核心机制**：每次 rollout 记录 notes，notes 就是 experience buffer。后续 agent 读取 notes 在 context 中调整行为——权重没变，策略变了
- **RL 粒度**：Task 是 rollout 单位，Expert 是多个 task context 并集的涌现结果
- **信息沉淀路径**：notes（task 细节）→ doc（跨 task 知识并集）→ SOP（结晶策略），且压缩过程本身也是 task，用同一套机制学习。系统只有一种原语：task rollout + notes 记录
- **两种模式**：
  - Human-driven：人在 loop 里做 verifier，meta-agent 已经实现得比较好
  - Autonomous：agent 自己跑 rollout，自己拿 outcome signal。这是 bfrl 要重点解决的
- **下一步待定义**（autonomous 模式）：
  1. Task 从哪来
  2. Outcome signal 从哪来（没有人的时候）
  3. Rollout loop 怎么跑

### Context 与基模亲和性

- 先聚焦单模型 autonomous 模式，基模无关性作为未来特性

### 特定领域 BFRL 研究

- 重新聚焦项目里尚未解决的 autonomous 问题：`task 从哪来`、`outcome signal 从哪来`、`rollout loop 怎么跑`
- 明确结论：特定领域 BFRL 不能理解成“往 prompt 里塞更多领域知识”，而必须把领域改造成一个可反复 rollout、可观测 outcome、可沉淀 context 的闭环
- 新增一个关键视角：BFRL 的核心难点不是权重 credit assignment，而是 **context credit assignment**
  - 一次 rollout 的经验到底应该写入 `memory/daily-notes/`
  - 还是提炼为 `memory/doc/`
  - 还是升级为 `memory/sop/`
  - 还是修订检索/上下文组装策略
  - 还是反过来说明 `outcome harness` 本身定义错了
- 明确了特定领域适合做 BFRL 的前提：
  1. task 重复出现且重叠度高
  2. agent 能真实操作领域环境
  3. outcome 可从环境中拿到
  4. 反馈周期不能太长
- 明确了 task source 至少要混合三类：
  - 真实任务
  - 失败重放任务
  - 围绕真实任务做局部扰动的合成任务
- 明确了 outcome signal 的设计原则：
  - 仍然坚持 outcome-based，不引入额外 verifier 模型
  - 但应设计成领域原生的 `outcome harness`
  - harness 可以输出 success / quality / cost / risk / escalation 等 outcome card
- 明确了 rollout loop 的本质不是单次问答，而是“上下文策略搜索”：
  1. 取 task
  2. 检索 notes/doc/SOP
  3. 组装本次 context
  4. agent 执行并调用工具
  5. 环境返回 outcome
  6. 写回经验
  7. 判断经验应该沉淀到哪一层
- 明确了特定领域 BFRL 的真正优化对象不是模型权重，而是：
  - task 检索
  - context 组装
  - 工具选择
  - SOP 选择
  - 经验沉淀路径
  - 人机升级边界
- 写出 `memory/doc/domain-bfrl.md`，把上述方法系统化

### 当前判断

- 真正可落地的路线不是“先做通用 BFRL 框架，再套领域”，而是“先选一个 outcome 清晰的窄领域，把它改造成 task loop + outcome harness + context compression 的闭环”
- 对这个项目来说，下一步最有价值的不是继续抽象讨论，而是选一个具体领域，定义：
  1. task schema
  2. outcome harness
  3. 最小 rollout loop

### 特定领域 BFRL 的关键点与策略

- 补充了一轮判断：要把特定领域 BFRL 做成，不是先追求“领域知识更多”，而是先确保这个领域具备闭环优化条件
- 当前认为最关键的几点是：
  1. 领域必须切得足够窄，保证 task 重复度和相似度足够高
  2. 必须优先定义 `outcome harness`，没有稳定 outcome 就没有真正的 RL
  3. 优化对象不是权重，而是 `task 取样`、`context 组装`、`工具选择`、`SOP 选择`、`经验沉淀` 和 `升级边界`
  4. 必须显式处理 **context credit assignment**，即一次 rollout 的经验应该进入 `note`、`doc`、`SOP`、检索策略还是 reward 设计修订
  5. notes 不能只是保存，还必须能被后续 rollout 有效检索和利用
- 当前建议采用的策略是：
  - `Outcome-first`：先做领域原生的 outcome harness
  - `Failure-first`：失败样本和边界 case 高优先级重放
  - `Replay-first`：混合真实任务、失败重放、局部扰动合成任务
  - `Compression-first`：尽早建立 `notes -> doc -> SOP` 的升降级规则
  - `Escalation-first`：预先定义哪些情况必须交给人
  - `Shadow-first`：先旁路运行，稳定后再放权
- 进一步收敛出的行动建议：
  1. 不再继续抽象讨论“通用领域 BFRL”
  2. 直接选一个窄领域做首个原型
  3. 先定义 `task schema`
  4. 再定义 `outcome harness`
  5. 最后接最小 `rollout loop`
